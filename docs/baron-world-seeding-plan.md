# Baron World Seeding Plan

This document defines how the existing `FugitiveBaron` Paper plugin should seed world content into an already-generated Minecraft world. The preferred approach is plugin-driven post-generation placement rather than native terrain generation.

## Goal

When the server owner runs one seeding command, the plugin should be able to:

- build or stamp `antenna_nest` at the active hideout
- place rumor boards near towns and settlements
- place a small number of remote clue sites
- populate notes, books, and themed containers
- remember what it has already placed so it does not duplicate everything on every restart

## Why Post-Generation Placement

This project already has:

- a Paper plugin
- a live existing world
- authored narrative content tied to plugin logic

So post-generation placement is the best fit because it:

- works on existing worlds
- keeps all Baron logic in one system
- lets us evolve content without reworking worldgen
- allows one-time seeded artifacts plus manual refinement later

## Seeding Model

Use a hybrid content model:

- structure-like placements are generated by plugin code
- authored text and inventories are applied after block placement
- future structure templates can replace code-built layouts later

Phase one should not attempt a full general structure templating engine. It should seed one hideout, one board type, and two clue site types reliably.

## Phase One Seed Targets

### 1. Antenna Nest

Seed exactly one authored hideout at the configured active hideout location.

What should be placed:

- shack shell
- antenna frame
- signal desk
- ledger corner
- liquor shelf
- sleep alcove
- burn pile
- concealed stash
- lectern with main Baron book
- themed containers

### 2. Rumor Boards

Seed one rumor board near every detected settlement.

Minimum target settlement types:

- vanilla villages
- any manually configured extra settlements later

### 3. Remote Clue Sites

Seed:

- one abandoned camp note site
- one Techtonic disposal box site

These can be chosen near roads, village outskirts, shorelines, or other sensible player-travel corridors.

## Plugin Responsibilities

The plugin should gain a dedicated seeding service rather than burying this logic inside `HideoutService` or `BaronCommand`.

Recommended new service:

- `WorldSeedService`

Responsibilities:

- choose seed targets
- detect existing placements
- place blocks
- populate containers and lecterns
- persist seed state
- expose admin commands

## Persistence

Seeding must be idempotent. If the server restarts, it should not duplicate boards or overwrite hand edits carelessly.

Persist state in the plugin data folder, for example:

- `plugins/FugitiveBaron/seed-state.yml`

Record:

- whether `antenna_nest` has been seeded
- coordinates of every seeded rumor board
- coordinates and ids of remote clue sites
- seed version number

### Seed Versioning

Use a simple integer version:

- `world-seed-version: 1`

If the seeding logic changes later, the plugin can:

- skip old seeded content
- report mismatch
- offer an explicit reseed command

## Settlement Detection

Phase one should keep this pragmatic.

### Primary Detection Method

Use Bukkit world structures to find nearby villages and settlement anchors around the world spawn and around explored areas if feasible.

Practical options:

- use `world.locateNearestStructure(...)` iteratively for villages
- or maintain a configured list of settlement centers in config

### Recommended Phase One Choice

Use a mixed model:

- auto-detect vanilla villages where possible
- support manual config for extra settlements

This avoids pretending the plugin can infer every custom town from terrain alone.

## Settlement Config Extension

Add a config section later such as:

```yaml
settlements:
  spawn_town:
    world: world
    x: 12
    y: 70
    z: 4
    board-style: village_well
  dockside:
    world: world
    x: 312
    y: 64
    z: -120
    board-style: dock
```

The plugin can merge:

- detected vanilla villages
- configured settlements

## Rumor Board Placement Rules

For each settlement center:

- offset the board a short distance from the center
- avoid water, leaves, and roofs
- prefer flat ground near paths, bells, wells, or gates
- do not place inside occupied villager beds or workstations

Placement search strategy:

1. start from settlement center
2. spiral or random-sample nearby blocks within `8-16` blocks
3. choose the first valid flat area
4. place the board facing the main approach

### Valid Board Footprint

Use a small footprint:

- `3x1` or `3x2`
- fence posts or stripped logs
- signs, hanging signs, trapdoors, lantern

This keeps placement robust.

## Antenna Nest Placement Rules

The active hideout location already exists in config. That should remain the single source of truth.

Phase one should:

- flatten a modest footprint if necessary
- preserve terrain as much as practical
- place the shack centered on the hideout location
- put the lookout side facing the best open view

Use a small terrain adaptation layer:

- find highest solid blocks in footprint
- raise low edges slightly with stilts, stone, or supports
- avoid giant terraforming

## Remote Clue Placement Rules

Remote clue sites do not need perfect procedural sophistication.

### Abandoned Camp

Place:

- near a path, riverbank, cave mouth, or village edge
- within moderate travel distance of spawn or a settlement

### Techtonic Disposal Box

Place:

- near a village outskirts path
- ruined shack
- dock
- roadside chest spot

Both should feel discoverable, not hidden like admin secrets.

## Content Population

After placing blocks, the plugin should populate:

- lecterns with written books
- barrels and chests with curated contents
- signs with selected rumor lines
- named paper items for note fragments

This should be code-driven from canonical text assets rather than hard-coded ad hoc strings spread across multiple classes.

Recommended future content class:

- `WorldContentLibrary`

It would supply:

- book text
- sign sets
- item names
- settlement board variants
- container loadouts

## Command Surface

Add explicit seed commands to the existing `/fugitivebaron` admin command.

Recommended subcommands:

- `/fugitivebaron seed world`
- `/fugitivebaron seed antenna`
- `/fugitivebaron seed boards`
- `/fugitivebaron seed clues`
- `/fugitivebaron seed status`

### Expected Behavior

- `seed world`
  runs the entire phase one seeding pass
- `seed antenna`
  places or refreshes only the Antenna Nest
- `seed boards`
  places settlement rumor boards only
- `seed clues`
  places remote clue sites only
- `seed status`
  reports what has already been seeded

## Safety Rules

The seeding system should be conservative.

- never overwrite large player-built areas silently
- skip placement if the footprint is heavily occupied
- report skipped locations to the command sender
- prefer "seed once, then hand-refine" rather than destructive regeneration

### Occupancy Heuristic

For phase one, treat a footprint as suspicious if it contains:

- chests not placed by the plugin
- many non-natural crafted blocks
- doors, beds, furnaces, or signs in dense arrangement

If suspicious:

- skip and warn
- allow a future force flag if needed

## Proposed Internal Classes

Recommended additions:

- `WorldSeedService`
- `SeedStateRepository`
- `WorldContentLibrary`
- `StructurePainter`
- `SettlementLocator`

### WorldSeedService

Coordinates the seeding pass and subcommands.

### SeedStateRepository

Reads and writes `seed-state.yml`.

### WorldContentLibrary

Owns note text, board line sets, and container recipes.

### StructurePainter

Places simple structures block-by-block from declarative definitions.

### SettlementLocator

Finds vanilla villages and merges configured settlement centers.

## Implementation Strategy

### Phase 1A

- implement persistence
- implement `seed status`
- implement `seed antenna` with block-by-block placement

### Phase 1B

- implement a simple rumor board structure
- add configured and detected settlement centers
- implement `seed boards`

### Phase 1C

- implement abandoned camp and disposal box clue sites
- implement `seed clues`

### Phase 1D

- add `seed world`
- add occupancy checks and better logging

## Relation To Existing Plugin

This seeding work should integrate with the current plugin rather than replace it.

Existing pieces to reuse:

- `HideoutService` for active hideout coordinates
- existing docs and authored note text
- `FugitiveBaronPlugin` as the service registry
- `BaronCommand` as the admin command entry point

Avoid:

- burying worldgen logic inside `HideoutService`
- scattering note text into command classes
- making seeding implicit on every startup before it is proven safe

## Recommended First Real Milestone

The first milestone should be narrow and demonstrable:

1. `/fugitivebaron seed antenna`
2. `/fugitivebaron seed boards`
3. `/fugitivebaron seed status`

That is enough to make the Baron feel embedded in the world without overcommitting to full procedural content immediately.
